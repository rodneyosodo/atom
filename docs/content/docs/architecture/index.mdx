## System overview

```
┌─────────────────────────────────────────────┐
│                   Atom                       │
│                                             │
│  ┌──────────────┐    ┌────────────────────┐ │
│  │   Identity   │    │   Authorization    │ │
│  │   Module     │    │   Module           │ │
│  │              │    │                    │ │
│  │ entities     │    │ resources          │ │
│  │ credentials  │    │ capabilities       │ │
│  │ sessions     │    │ roles              │ │
│  │ groups       │    │ policy_bindings    │ │
│  │ ownerships   │    │ PDP engine         │ │
│  └──────────────┘    └────────────────────┘ │
│                                             │
│  ┌─────────────────────────────────────────┐│
│  │          PostgreSQL (single DB)         ││
│  └─────────────────────────────────────────┘│
└─────────────────────────────────────────────┘
```

Atom is a single HTTP service backed by a single Postgres database. There are no external dependencies, no message queues, no caches required for correctness.

## Module structure

### Identity module (`src/identity/`)

Manages all principals and their authentication:

- **Entities** — the universal principal type. Kind values: `human`, `device`, `service`, `workload`, `application`.
- **Credentials** — password (argon2-hashed) and API key (format `atom_<id>_<secret>`, secret argon2-hashed).
- **Sessions** — created on login, referenced by the JWT `sid` claim. Revokable.
- **Groups** — named collections of entities, used as policy subjects.
- **Ownerships** — parent–child relationships between entities.

### Authorization module (`src/authz/`)

Manages protected resources and evaluates access:

- **Resources** — anything protected by authorization: channels, devices, workspaces, secrets.
- **Capabilities** — atomic permission names (`read`, `write`, `publish`, `subscribe`, `execute`, `manage`, `delete`).
- **Roles** — named bundles of capabilities.
- **Policy bindings** — the core authorization primitive, binding a subject to a grant over a scope.
- **PDP engine** (`src/authz/engine.rs`) — evaluates `POST /authz/check` requests.

## Layered architecture

Every request flows through three layers:

```
HTTP Request
     │
     ▼
┌─────────────┐
│   Handler   │  – HTTP concerns only: extract params, call service/engine, return response
└──────┬──────┘
       │
       ▼
┌─────────────┐
│Service/Engine│ – Business logic, validation, audit writes
└──────┬──────┘
       │
       ▼
┌─────────────┐
│    Repo     │  – SQL queries only, returns domain types
└─────────────┘
```

Handlers never touch SQL. Repos never contain business logic. Cross-cutting concerns (auth extraction, error mapping) live in `src/auth.rs` and `src/error.rs`.

## AppState

```rust
pub struct AppState {
    pub pool:   PgPool,
    pub config: Config,
    pub keys:   Arc<RwLock<ActiveKeys>>,
}
```

`AppState` is cheaply cloned into every Axum handler via the `State` extractor. The `pool` is an Arc-wrapped connection pool internally. `keys` is an `Arc<RwLock<ActiveKeys>>` — the live signing key store, updated in-place on key rotation without a restart.

## Online authorization

Tokens in Atom carry **no permissions**. A JWT proves identity (who) and references a session (revocable). Authorization (whether an entity may perform an action on a resource) is always evaluated at runtime by the PDP.

This means:
- Policy changes take effect immediately — no token reissuance needed.
- Revoking a session immediately invalidates all tokens issued for it.
- External services can check authorization by calling `POST /authz/check` rather than decoding token claims.

## Audit log

Every significant event (login attempt, credential creation, authorization decision) writes a row to `audit_logs`. The write is fire-and-forget — a failure never blocks the primary operation.
