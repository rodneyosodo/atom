Atom is a single-binary replacement for Keycloak тАФ an HTTP service that combines identity management and policy-based authorization in one deployable unit.

Built for [Magistrala](https://github.com/absmach/magistrala), but generic enough for any cloud-native or edge system.

## What it does

| Concern | What Atom provides |
|---|---|
| **Identity** | CRUD for any principal type тАФ humans, devices, services, workloads, applications. All are first-class _entities_. |
| **Authentication** | Password login returning a JWT, long-lived API keys, session management with revocation. |
| **Authorization** | Policy-based decision engine (PDP) supporting RBAC, ABAC, and hybrid policies evaluated at runtime. |
| **Grouping** | Entities belong to groups; policies apply to groups. |
| **Ownership** | ParentтАУchild relationships between entities. |
| **Multi-tenancy** | All resources are tenant-scoped. |

## GraphQL

Atom exposes GraphQL at `POST /graphql`; debug builds also expose `/graphql/playground`. GraphQL uses the same Bearer token authentication as REST. The schema covers health, login/logout/session lookup, tenants, profiles, profile versions, entities, resources, groups, credentials, ownerships, roles, capabilities, policies, authz checks, audit logs, and profile-driven entity creation. REST remains available and unchanged.

For entity creation, frontends should send `profileId` when a profile is selected. Atom derives the internal `kind` from `profile.kind`; if `profileVersionId` is omitted, Atom uses the active/latest profile version. Authorization still uses `entities.kind`, not `profileId` or `profileVersionId`.

```graphql
mutation {
  login(input: {
    identifier: "atom-admin",
    secret: "change-me",
    kind: "password"
  }) {
    token
    entityId
    sessionId
    expiresAt
  }
}

mutation {
  createTenant(input: {
    name: "factory-a",
    route: "factory-a"
  }) {
    id
    name
    route
    status
  }
}

mutation {
  createEntity(input: {
    profileId: "client-profile-id",
    name: "meter-001",
    attributes: {
      serial_no: "WM-001"
    }
  }) {
    id
    kind
    profileId
    profileVersionId
    attributes
  }
}

mutation {
  createResource(input: {
    kind: "channel",
    name: "telemetry",
    attributes: {
      topic: "telemetry"
    }
  }) {
    id
    kind
    name
    attributes
  }
}

mutation {
  createPolicy(input: {
    subjectKind: "entity",
    subjectId: "client-entity-id",
    grantKind: "capability",
    grantId: "publish-capability-id",
    scopeKind: "object",
    scopeRef: "channel-resource-id",
    effect: "allow"
  }) {
    id
    effect
  }
}

mutation {
  authzCheck(input: {
    subjectId: "client-entity-id",
    action: "publish",
    resourceId: "channel-resource-id"
  }) {
    allowed
    reason
  }
}
```

Magistrala mapping uses generic Atom operations: domains call `createTenant`, clients call `createEntity` with `kind=device` through the `client` profile, and channels call `createResource` with `kind="channel"`.

## Core design principles

**Entity-first** тАФ everything is an entity. There is no special "user" class. Humans, devices, and services are all modeled identically and treated symmetrically by the authorization engine.

**Online authorization** тАФ tokens carry no embedded permissions. Every authorization decision calls the PDP at runtime. This means policy changes take effect immediately, without token reissuance.

**Deny by default** тАФ an explicit `DENY` policy binding overrides any `ALLOW`. No matching allow policy means denied.

**Single binary** тАФ one Go binary, one Postgres database. No sidecars, no service mesh required.
